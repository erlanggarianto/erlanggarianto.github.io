Overview

The Short Version

Suarise is an Indonesian company focused on digital accessibility. They were building a new platform where differently-abled users could report accessibility issues on websites and apps, and developers could learn how to fix them. I was brought in as a freelance UX designer to build the design system from the ground up.

During early reviews, I found that the platform needed an accessibility-first foundation before any features could be built on top of it. I studied existing accessibility-focused tools and design systems, then built a complete system covering typography, color, icons, and 24 reusable components, all grounded in WCAG standards. We tested the sign-up and onboarding flows with users who have low to no vision, found real usability issues that compliance checklists alone would not have caught, and iterated until the flows worked.

Context

Context

What is Suarise?

Suarise is an Indonesian social enterprise focused on digital accessibility. They wanted to build a platform that serves two purposes: let differently-abled users report accessibility problems they encounter on websites and apps, and provide resources for developers to learn how to fix those problems. The platform was being built from scratch, so there was no existing product or design system to work from.

Why this project mattered

An accessibility platform that is not itself accessible would be a contradiction. The design system had to be built with WCAG compliance as a foundation, not as an afterthought. Every component, every color choice, every interaction pattern needed to work for users with disabilities, including users with low to no vision who rely on screen readers.

My role

I was the sole UX designer on this freelance project, working with the Suarise director, a UX researcher, a UX writer, and frontend engineers. I handled user and competitor research, accessibility standards and design principles, component design and documentation, and prototyping and testing with differently-abled users. The project took about one month.

Research

Research: Understanding the Space

I started by studying two things: accessibility-focused tools that already existed, and established design systems that handled accessibility well.

Competitor analysis

I looked at tools like ARC Toolkit and axe DevTools to understand how they designed for feedback and reporting. Most of these tools were very technical and built for developers, not for everyday users. Since Suarise wanted their platform to be usable by non-developers as well, I aimed to create something that was approachable while still being fully compliant.

🖼 Competitive Analysis ARC Toolkit, axe DevTools, and Fable

Design systems research

I studied Google's Material Design and Apple's Human Interface Guidelines to understand how they balanced accessibility with aesthetics. Material Design had strong scalability and contrast systems. Apple's HIG focused on clarity and visual hierarchy. I combined ideas from both to create a system that felt modern and structured while meeting accessibility requirements.

Design System

Building the Design System

Deciding on WCAG compliance levels

Before building any components, we needed to agree on which WCAG compliance standards to follow for each type of element. This was not as straightforward as it sounds.

The team went back and forth on this. Everyone had their own opinion on what mattered and what did not. For example, some elements had multiple WCAG criteria that could apply, and not all of them were equally important for our specific use case. We discussed each element type (buttons, toggles, form fields, etc.) and tried to reach agreement on which compliance level to target and which specific criteria to prioritize.

This process was important because blindly applying every WCAG rule without discussion would have led to an overly rigid system. Instead, we made intentional choices about where to be strict and where to be pragmatic, always with our actual users in mind.

Accessibility standards check

I audited the kinds of UI elements we would need (buttons, forms, text, navigation) against WCAG criteria for contrast and hierarchy. I created a checklist to evaluate each element, which helped the team identify what standards each component needed to meet.

🖼 WCAG Audit Checklist Element compliance mapping

Typography

I defined a clear type hierarchy with readable type scales. Each level (Display, Heading, Body, Caption) had specific size, line-height, and weight values, with usage guidelines for where each should be used (hero sections, page titles, body content, metadata, etc.).

🖼 Typography Scale Display, Heading, Body, and Caption levels with specs

Color

I built a contrast-safe color palette based on WCAG standards. Every color combination was tested using Stark to verify it met minimum contrast ratios. The palette included semantic colors (red for errors, green for success, yellow for warnings, blue for information) and a gray scale ranging from primary text to secondary surfaces.

🖼 Color Palette Contrast ratios and semantic color definitions

Icons

I designed simple icon shapes that remained readable at any size, important for users with low vision who may be using magnification.

Components

I built 24 reusable components, including buttons, toggles, checkboxes, radio buttons, text fields, tips, banners, bottom sheets, cards, and calendar elements. Each component included:

  • Multiple states (default, hover, active, disabled, error)
  • Focus states for keyboard navigation
  • ARIA labels for screen reader compatibility
  • Sticker sheets showing all variants
  • Overview, purpose, and usage guidelines

Usability Testing

Testing with Differently-Abled Users

When I presented the first version of the design system to the team, engineers requested real-world validation beyond just compliance checklists. So we built a sign-up and onboarding flow prototype and tested it with users who have low to no vision.

The testers were recruited through the Suarise director's existing network. She had been working in the disability space for a while and had conducted many activities with this community. The testers were already familiar with Suarise and what the organization does.

What we tested

We focused specifically on the authentication and onboarding flow, which included account creation, personal information forms, and initial setup steps. This was the first experience new users would have with the platform, so it had to work.

What we found

Testing revealed three issues that compliance checklists alone would not have caught:

1
Too many form fields.The onboarding forms asked for a lot of personal information at once. For screen reader users, long forms are especially tiring because they have to navigate through each field sequentially. What felt like a normal form for sighted users felt endless for screen reader users.
2
Screen readers could not read nested labels.Some form fields used nested label structures that sighted users could parse visually, but screen readers could not interpret correctly. Users heard incomplete or confusing label text, making it unclear what information was being asked for.
3
Too much content on one page.The onboarding packed too much information onto a single screen. For screen reader users, this meant listening to a lot of content before reaching the actual interactive elements. The cognitive load was too high.
🖼 Usability Testing Sign-up and onboarding flow with screen reader users — before and after

What I changed

I worked with engineers to address each issue:

  • Broke the onboarding into shorter, focused steps so each screen had fewer fields and less content.
  • Simplified the layout and improved navigation between steps.
  • Fixed ARIA labeling so screen readers could correctly read every form field and its purpose.

After implementing the changes, users completed the sign-up flow faster and made fewer mistakes.

Result

The Result

The design system gave Suarise a solid, accessible foundation to build their platform on. It standardized typography, colors, icons, and 24 components, helping the team design and build faster while ensuring inclusivity for all users, including those with disabilities.

The system was documented with usage guidelines for each component, making it possible for engineers to implement components consistently without needing to consult me on every detail.

Reflection

Reflection

What went well

  • Testing with real users who have low to no vision was the most valuable part of this project. It showed us problems that no amount of WCAG checklist compliance would have revealed. Accessibility is not just about meeting technical criteria. It is about whether real people can actually use the product.
  • The team discussions about which WCAG compliance levels to target for each element were productive. Instead of blindly following every rule, we made intentional decisions about where to be strict and where to be practical. This gave us a system that was genuinely accessible without being unnecessarily rigid.
  • Building the design system from scratch meant there was no legacy baggage to work around. Every decision was made with accessibility in mind from the start, which is much easier than retrofitting an existing system.

What I would do differently

  • Introduce accessibility testing earlier in the processWe built most of the components before testing with real users. If we had tested earlier with even rough prototypes, we could have caught the label and layout issues sooner and saved time on rework.
  • Strengthen the feedback loop between design and engineeringSome accessibility issues only showed up when components were implemented in code, because the way a screen reader interprets HTML can differ from what you see in a Figma prototype. Closer collaboration during implementation would have caught these faster.
  • Add more detailed accessibility annotations to each componentThe current documentation covered states, guidelines, and ARIA labels, but more specific notes about keyboard navigation patterns, focus order, and screen reader behavior for each component would have made the engineering handoff even smoother.
